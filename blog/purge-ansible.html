Automatiser la purge applicative avec Shell et Ansible

Mars 2026 • Retour d'expérience — Alternance CTI (Assurance Maladie)

BashAnsibleLinuxAutomatisation

Le problème

Au CTI (Centre de Traitement Informatique) de l'Assurance Maladie à Saint-Étienne, les serveurs de production accumulent des versions obsolètes d'applications au fil des déploiements. Le problème : chaque serveur peut contenir des dizaines de versions inutilisées qui consomment de l'espace disque précieux, sans aucune valeur opérationnelle.

Avant mon intervention, la purge était manuelle : un technicien se connectait sur chaque serveur, identifiait les versions à supprimer, vérifiait qu'elles n'étaient plus actives, et les supprimait une par une. Temps moyen : 30 minutes à 1 heure par serveur. Avec ~300 serveurs concernés, la tâche était irréaliste manuellement.

La solution

J'ai conçu une solution en deux couches :

1. Script Shell — la logique métier

Un script Bash qui s'exécute localement sur chaque serveur et gère toute la logique :

  • Détection automatique des versions obsolètes (comparaison avec la version active en production)
  • Gestion des anomalies : si une version est dans un état incohérent (fichier verrou, processus actif), elle est signalée mais pas supprimée
  • Logs détaillés : chaque action est tracée (version supprimée, espace libéré, anomalies rencontrées)
  • Mode dry-run : possibilité de simuler la purge sans rien supprimer pour validation
  • Rollback : sauvegarde avant suppression pour permettre un retour arrière si nécessaire

2. Playbook Ansible — l'orchestration

Un playbook Ansible qui déploie et exécute le script sur l'ensemble du parc :

  • Inventaire dynamique : ciblage par groupes de serveurs (prod, préprod, etc.)
  • Déploiement du script : copie automatisée sur chaque cible
  • Exécution orchestrée : parallélisation contrôlée (forks) pour limiter la charge
  • Collecte des rapports : récupération centralisée des logs de chaque serveur
  • Idempotence : le playbook peut être rejoué sans risque (les versions déjà purgées sont ignorées)

Les défis techniques

Gestion des cas limites

Le plus gros défi n'était pas la suppression en elle-même, mais la gestion des anomalies. En production, on rencontre des situations imprévues : versions partiellement déployées, fichiers verrouillés, liens symboliques cassés, etc. Le script devait être défensif par défaut : en cas de doute, ne rien supprimer et remonter l'information.

Sécurité et traçabilité

Dans un environnement de production lié à l'Assurance Maladie, chaque action doit être tracée et justifiable. Le système de logs permet de reconstituer exactement ce qui a été fait, quand, et sur quel serveur. Le mode dry-run permet de valider le comportement avant toute action réelle.

Le script a été validé en préproduction puis déployé progressivement sur le parc. L'exécution complète sur ~300 serveurs prend désormais quelques heures au lieu de semaines de travail manuel.

Ce que j'en retiens

  • Automatiser, c'est d'abord comprendre : avant d'écrire une seule ligne, j'ai passé du temps à comprendre le process manuel, les cas limites, les contraintes de production.
  • La sécurité avant la vitesse : le mode dry-run et le rollback n'étaient pas optionnels. En production, on ne prend pas de risques.
  • Ansible pour l'échelle : un bon script local + Ansible pour l'orchestration = une combinaison redoutable pour gérer un parc de serveurs.
  • Les logs, c'est la confiance : des logs clairs et complets permettent de valider le travail et de rassurer les équipes.
