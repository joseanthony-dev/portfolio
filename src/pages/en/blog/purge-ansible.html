Automating application purge with Shell and Ansible

March 2026 • Experience report — Work-study at CTI (French Health Insurance)

BashAnsibleLinuxAutomation

The Problem

At the CTI (Centre de Traitement Informatique) of the French Health Insurance in Saint-Etienne, production servers accumulate obsolete application versions over successive deployments. The issue: each server can contain dozens of unused versions consuming precious disk space, with no operational value.

Before my intervention, the cleanup was manual: a technician would connect to each server, identify versions to remove, verify they were no longer active, and delete them one by one. Average time: 30 minutes to 1 hour per server. With ~300 servers concerned, the task was unrealistic to perform manually.

The Solution

I designed a two-layer solution:

1. Shell Script — business logic

A Bash script that runs locally on each server and handles all the logic:

  • Auto-detection of obsolete versions (comparison with the active production version)
  • Anomaly handling: if a version is in an inconsistent state (lock file, active process), it is flagged but not deleted
  • Detailed logs: every action is traced (deleted version, freed space, encountered anomalies)
  • Dry-run mode: ability to simulate the purge without deleting anything for validation
  • Rollback: backup before deletion to allow reverting if needed

2. Ansible Playbook — orchestration

An Ansible playbook that deploys and executes the script across the entire fleet:

  • Dynamic inventory: targeting by server groups (prod, pre-prod, etc.)
  • Automated script deployment: automated copy to each target
  • Controlled parallel execution: controlled parallelization (forks) to limit load
  • Centralized report collection: centralized retrieval of logs from each server
  • Idempotence: the playbook can be replayed safely (already purged versions are skipped)

Technical Challenges

Edge case handling

The biggest challenge was not the deletion itself, but anomaly handling. In production, you encounter unexpected situations: partially deployed versions, locked files, broken symlinks, etc. The script had to be defensive by default: when in doubt, delete nothing and report the information.

Security and traceability

In a production environment tied to the French Health Insurance, every action must be traced and justifiable. The logging system allows reconstructing exactly what was done, when, and on which server. The dry-run mode allows validating behavior before any real action.

Key Takeaways

  • Automating starts with understanding: before writing a single line, I spent time understanding the manual process, edge cases, and production constraints.
  • Safety before speed: dry-run mode and rollback were not optional. In production, you don't take risks.
  • Ansible for scale: a good local script + Ansible for orchestration = a powerful combination for managing a server fleet.
  • Logs build trust: clear and comprehensive logs allow validating the work and reassuring the teams.
